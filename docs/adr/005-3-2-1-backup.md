# ADR-005: 3-2-1 backup strategy

## Статус
Accepted

## Контекст
Правило 3-2-1 (3 копии, 2 носителя, 1 offsite) — индустриальный стандарт.
На VPS критичны: конфиги (/etc), данные приложений (/var/www), пользовательские файлы (/home), а также /var/log и /root.

Требования:
- Автоматический daily backup
- Encrypted at rest (AES-256)
- Deduplication (экономия места)
- Snapshot-based (точечное восстановление)
- Минимум зависимостей

## Решение
**Инструмент: restic** — AES-256-GCM, dedup, snapshots, rclone для Yandex Disk, s3 backend.

**Три копии:**
1. **Локальная** — `/var/backups/cloud.ru-free-tier-vm` (restic local)
2. **cloud.ru S3** — restic s3 backend, AES-256
3. **Yandex Disk (offsite)** — restic + rclone, географически отдельный дата-центр

**Retention:** 7 daily / 4 weekly / 3 monthly / 1 yearly (настраивается в `config/backup.yaml`)

**Расписание:** cron `0 2 * * *` (ночь, минимальная нагрузка)

**Конфигурация** (`config/backup.yaml`):
```yaml
sources:
  - /etc
  - /home
  - /var/www
  - /var/log
  - /root
retention:
  keep_daily: 7
  keep_weekly: 4
  keep_monthly: 3
  keep_yearly: 1
```

**Команды:** `create`, `list`, `restore`, `setup`, `status` (все через `scripts/backup.py`)

## Альтернативы
- **Borg**: медленнее restic на больших данных, нет встроенного S3/s3 поддержки
- **Duplicity**: инкрементальный, но dated (gpg-based, нет dedup repo-stats)
- **rsync + сценарии**: не snapshot-based, нет dedup, сложный restore
- **tar + gpg + s3cmd**: никакого управления, ручной restore
- **Bareos**: enterprise, тяжеловесен для single VPS

## Последствия
- Ежедневный backup: ~8 MB/day (сжатие + dedup, зависит от изменений)
- Восстановление: `python3 scripts/backup.py restore --snapshot <id>`
- Все 3 копии зашифрованы одним `restic/password` из BSM/.env
- Очистка старых snapshot-ов автоматическая (`forget --prune`)
- Cron настроен через `scripts/backup.py setup` (добавляет строку в crontab)
- Лог backup: `/var/log/backup.log`
